# Домашнее задание: JWT + Spring Security

## Тема

Практика по аутентификации и авторизации в Spring Boot приложении с использованием **JWT (JSON Web Tokens)** и **Spring Security**.

## Проект: «Подземелье драконов» (Dragon's Lair API)

Вам предстоит реализовать REST API для онлайн-игры в жанре текстового RPG, где игроки исследуют подземелье, сражаются с монстрами и собирают сокровища. Часть API доступна всем (например, просмотр бестиария и таблицы лидеров), а часть — только зарегистрированным игрокам, мастерам подземелий и администраторам.

> Если тема показалась скучной — можно заменить на свою, **но согласовать с преподавателем**. Главное, чтобы сохранились требования к функционалу (см. ниже). Примеры альтернатив: API для аукциона лотов, сервис бронирования квестов, мини-соцсеть для мемов, бэкенд для покерного клуба и т.п. Тудулисты и онлайн-библиотеки — **запрещены**.

## Требования к функционалу

### 1. Регистрация и аутентификация

- `POST /api/auth/register` — регистрация нового игрока (логин, пароль, email).
- `POST /api/auth/login` — вход, возвращает **access token (JWT)** и **refresh token**.
- `POST /api/auth/refresh` — обновление access-токена по refresh-токену.
- Пароли должны храниться в БД в **хешированном** виде (`BCryptPasswordEncoder`).
- JWT должен содержать `username`, `roles`, `exp`, `iat`.

### 2. Роли пользователей

Минимум **3 роли**:

| Роль          | Описание |
|---------------|----------|
| `ROLE_PLAYER` | Обычный игрок. Может сражаться, собирать лут, смотреть свой профиль. |
| `ROLE_MASTER` | Мастер подземелья. Может создавать новых монстров и подземелья. |
| `ROLE_ADMIN`  | Администратор. Может банить игроков, удалять любые сущности, выдавать роли. |

Управление ролями должно быть реализовано через `GrantedAuthority` и проверяться через `@PreAuthorize` или конфигурацию `SecurityFilterChain`.

### 3. Маршруты

#### Публичные (без авторизации)

- `GET /api/bestiary` — список всех монстров.
- `GET /api/bestiary/{id}` — карточка монстра.
- `GET /api/leaderboard` — топ-10 игроков по опыту.
- `POST /api/auth/**` — регистрация / вход / refresh.

#### Защищённые (только авторизованные)

- `GET /api/me` — профиль текущего игрока.
- `POST /api/dungeons/{id}/enter` — войти в подземелье.
- `POST /api/dungeons/{id}/attack` — атаковать монстра.
- `GET /api/inventory` — инвентарь игрока.

#### Только для `ROLE_MASTER` и выше

- `POST /api/bestiary` — создать монстра.
- `POST /api/dungeons` — создать подземелье.
- `PUT /api/bestiary/{id}` — отредактировать монстра.

#### Только для `ROLE_ADMIN`

- `DELETE /api/players/{id}` — забанить (удалить) игрока.
- `POST /api/players/{id}/roles` — выдать/забрать роль.
- `DELETE /api/bestiary/{id}` — удалить монстра.

### 4. Безопасность

- Все защищённые маршруты должны проверять валидность JWT через **кастомный фильтр** (`OncePerRequestFilter`).
- При невалидном/просроченном токене — статус `401 Unauthorized`.
- При недостаточных правах — статус `403 Forbidden`.
- CSRF можно отключить (REST API stateless).
- Сессии — `SessionCreationPolicy.STATELESS`.
- Глобальный `@ControllerAdvice` для обработки ошибок безопасности.

### 5. Доменная модель (минимум)

- `User` — id, username, password, email, roles, experience.
- `Role` — id, name.
- `Monster` — id, name, hp, attack, level, lootTable.
- `Dungeon` — id, name, difficulty, monsters.
- `Item` — id, name, rarity, ownerId.

## TDD-подход и автотесты

В проекте уже лежат **интеграционные тесты** (`src/test/java/com/example/demo/integration/`), которые проверяют корректность вашей реализации. Сейчас они **красные** — это нормально, это и есть TDD: ваша задача — написать код так, чтобы все тесты позеленели.

Тесты поднимают **настоящий PostgreSQL** через [Testcontainers](https://testcontainers.com/) — никакого H2. Поэтому для запуска тестов локально нужен установленный и запущенный **Docker**.

Запуск тестов:

```bash
./mvnw verify
```

Что покрывают тесты:

- `AuthFlowIntegrationTest` — регистрация, логин, refresh, хеширование паролей, обработка дубликатов и неверных кредов.
- `PublicRoutesIntegrationTest` — публичные маршруты (`/api/bestiary`, `/api/leaderboard`) доступны без токена.
- `ProtectedRoutesIntegrationTest` — `/api/me` и `/api/inventory` требуют валидный JWT, без него → `401`, с битым → `401`.
- `RoleBasedAccessIntegrationTest` — `PLAYER` не может создавать монстров (`403`), `MASTER` может, `ADMIN` может банить игроков, `MASTER` — нет.

Для тестов ролей в БД при старте приложения должны быть **сидированы** демо-аккаунты (укажите способ их создания в README репозитория):

| username | password   | роль          |
|----------|------------|---------------|
| `master` | `masterPass` | `ROLE_MASTER` |
| `admin`  | `adminPass`  | `ROLE_ADMIN`  |

## CI / GitHub Actions

В проекте настроен workflow `.github/workflows/ci.yml`, который при каждом push и pull request:

1. Поднимает JDK 25.
2. Запускает `./mvnw verify` (интеграционные тесты с Testcontainers + PostgreSQL).
3. Сохраняет отчёты Surefire/Failsafe как артефакты.

**Сдача задания принимается только при зелёном CI**. Если в вашем репозитории workflow упал — задание не зачитывается.

## Как сдавать (форк-флоу)

Сдача ДЗ — **только через форк** этого репозитория. Никаких архивов, гугл-дисков и копи-паста в чат.

1. Сделайте **fork** этого репозитория в свой аккаунт на GitHub (кнопка **Fork** справа сверху).
2. Склонируйте свой форк локально:
   ```bash
   git clone https://github.com/<ваш-логин>/<имя-форка>.git
   cd <имя-форка>
   ```
3. Работайте в **отдельной ветке**, например `feature/jwt-security`:
   ```bash
   git checkout -b feature/jwt-security
   ```
4. Коммитьте небольшими логичными шагами с осмысленными сообщениями (`feat: add JWT filter`, `test: green auth flow`, и т.д.). Один гигантский коммит «всё сразу» — минус балл.
5. Пушьте в свой форк:
   ```bash
   git push origin feature/jwt-security
   ```
6. Убедитесь, что **GitHub Actions в вашем форке зелёный** (вкладка *Actions*). Сдача с красным CI не принимается.
7. Откройте **Pull Request из своей ветки форка в `main` исходного репозитория** (base repo — этот, head repo — ваш форк).
8. В описании PR укажите:
   - ФИО и группу,
   - краткое описание реализации,
   - ссылку на Swagger / Postman-коллекцию,
   - демо-аккаунты (если они отличаются от указанных в README).

> **Не мерджите PR сами.** Преподаватель проверит, оставит ревью и решит, что делать с веткой.

## Что должно быть в форке

1. Рабочий код (все тесты зелёные).
2. **README** в вашем форке с инструкцией по запуску (как поднять БД, какие переменные окружения нужны, как создаются демо-аккаунты).
3. **Postman / Insomnia коллекция** или OpenAPI/Swagger UI с описанием всех эндпоинтов.
4. **Демо-аккаунты**: по одному пользователю на каждую роль (логины/пароли — в README).
